# 🧩 Problem #1345 – Jump Game IV

**Difficulty:** Hard    
**Topics:** `Array`, `Hash Map`, `Breadth First Search`, `Graph`    
**Link:** [https://leetcode.com/problems/jump-game-iv/](https://leetcode.com/problems/jump-game-iv/)

---

## 📝 Problem Summary

You are given an integer array `arr`.

You start at index:

```txt
0
```

In one move, you may jump from index `i` to:

* `i + 1`
* `i - 1`
* any index `j` where:

```txt
arr[i] == arr[j]
```

Your goal is to reach the last index using the minimum number of jumps.

---

### Example

```txt
arr = [100,-23,-23,404,100,23,23,23,3,404]
```

Optimal path:

```txt
0 -> 4 -> 3 -> 9
```

Answer:

```txt
3
```

---

### Constraints

* `1 <= arr.length <= 5 * 10^4`
* `-10^8 <= arr[i] <= 10^8`

---

## 💡 Intuition

At first glance, this problem may look like:

* Dynamic Programming
* DFS + Memoization

because we want:

```txt
minimum jumps
```

However, this is actually a:

```txt
Shortest Path problem
```

which strongly suggests:

```txt
Breadth-First Search (BFS)
```

---

## 🔥 Key Recognition Pattern

Whenever a problem asks for:

* minimum steps
* minimum jumps
* minimum operations
* shortest path

and:

```txt
every move costs exactly 1
```

BFS is usually the correct approach.

---

## ❌ Why DFS + DP Fails

Suppose we define:

```txt
F(i) = minimum jumps from index i
```

The graph contains:

* cycles
* bidirectional movement
* teleport edges

Example:

```txt
i -> j
j -> i
```

Now recursive dependencies become cyclic.

This breaks the clean DAG structure typically required for DP.

---

## 🧠 Graph Interpretation

Treat each index as a graph node.

From index `i`, edges exist to:

* `i - 1`
* `i + 1`
* all indices having same value

Now the problem becomes:

```txt
Find shortest path
from node 0 to node n-1
```

This is classic BFS.

---

## 🛠 Approach — BFS

### Step 1 — Build Teleport Map

Store:

```txt
value -> list of indices
```

Example:

```txt
23 -> [5,6,7]
```

This allows O(1) access to all teleport destinations.

---

### Step 2 — Start BFS

Queue stores:

```txt
[index, steps]
```

We begin from:

```txt
[0, 0]
```

meaning:

* currently at index `0`
* used `0` jumps

---

### Step 3 — Explore Neighbors

For every index:

possible neighbors are:

```txt
index - 1
index + 1
same-value indices
```

Unvisited neighbors are pushed into BFS queue.

---

## 🔥 Critical BFS Rule

Visited nodes must be marked:

```txt
WHEN PUSHING into queue
```

NOT:

```txt
when removing from queue
```

Otherwise duplicate nodes may enter queue many times.

---

## 🚨 Important Optimization

Suppose:

```txt
arr = [7,7,7,7,7,7]
```

Without optimization:

every visited `7` re-traverses all other `7` indices.

This causes:

```txt
O(n²)
```

behavior.

---

### Optimization Trick

After processing teleport neighbors once:

```ts
teleportMap.delete(arr[index])
```

This guarantees:

```txt
Each teleport list is processed only once.
```

This is the key optimization that makes the solution efficient.

---

## 🚀 Queue Implementation Note (JavaScript)

Using:

```ts
queue.shift()
```

is inefficient because:

```txt
Array.shift() = O(n)
```

Instead:

* use normal array
* maintain a front pointer

Example:

```ts
let front = 0;
const item = queue[front++];
```

This provides:

```txt
O(1)
```

queue operations.

---

## ✅ Why BFS Guarantees Minimum Jumps

BFS explores nodes:

```txt
level by level
```

Meaning:

* all nodes reachable in 1 jump
* then all nodes reachable in 2 jumps
* then all nodes reachable in 3 jumps

So the first time we reach:

```txt
n - 1
```

we are guaranteed to have found the shortest path.

---

## 💻 Code (TypeScript)

```ts
function minJumps(arr: number[]): number {

    const n = arr.length;

    if (n === 1) {
        return 0;
    }

    // value -> list of indices
    const teleportMap = new Map<number, number[]>();

    for (let i = 0; i < n; i++) {

        if (!teleportMap.has(arr[i])) {
            teleportMap.set(arr[i], []);
        }

        teleportMap.get(arr[i])!.push(i);
    }

    const visited = new Array(n).fill(false);

    const queue: [number, number][] = [];

    // [index, steps]
    queue.push([0, 0]);

    visited[0] = true;

    let front = 0;

    while (front < queue.length) {

        const [index, steps] = queue[front++];

        // reached target
        if (index === n - 1) {
            return steps;
        }

        // index - 1
        if (
            index - 1 >= 0 &&
            !visited[index - 1]
        ) {
            visited[index - 1] = true;
            queue.push([index - 1, steps + 1]);
        }

        // index + 1
        if (
            index + 1 < n &&
            !visited[index + 1]
        ) {
            visited[index + 1] = true;
            queue.push([index + 1, steps + 1]);
        }

        // teleport neighbors
        const teleportIndexes =
            teleportMap.get(arr[index]) || [];

        for (const nextIndex of teleportIndexes) {

            if (!visited[nextIndex]) {

                visited[nextIndex] = true;

                queue.push([
                    nextIndex,
                    steps + 1
                ]);
            }
        }

        // critical optimization
        teleportMap.delete(arr[index]);
    }

    return -1;
}
```

## ⚖️ DFS vs BFS Comparison

| Approach | Suitable? | Reason                                    |
| -------- | --------- | ----------------------------------------- |
| DFS + DP | ❌         | Cyclic graph breaks clean DP dependencies |
| BFS      | ✅         | Finds shortest path in unweighted graph   |

---

## ⏱ Complexity Analysis

| Operation          | Complexity |
| ------------------ | ---------- |
| Build teleport map | `O(n)`     |
| BFS traversal      | `O(n)`     |
| Total Time         | `O(n)`     |
| Auxiliary Space    | `O(n)`     |

---

## 📌 Key Takeaways

* Minimum steps problems often map to BFS.

* Arrays can secretly represent graphs.

* BFS guarantees shortest path in unweighted graphs.

* Mark nodes visited when inserting into queue.

* Dynamic neighbor generation is often better than building full graph explicitly.

* Deleting processed teleport lists is the key optimization.

---

## ⚠️ Common Mistakes

### 1️⃣ Using DFS For Shortest Path

DFS does not naturally guarantee minimum steps.

---

### 2️⃣ Marking Visited Too Late

Visited nodes should be marked:

```txt
before pushing into queue
```

Otherwise queue size may explode.

---

### 3️⃣ Building Explicit Full Graph

Constructing all adjacency lists explicitly may consume huge memory.

Generate neighbors dynamically during BFS.

---

### 4️⃣ Forgetting Teleport Optimization

Without:

```ts
teleportMap.delete(arr[index])
```

same teleport lists are repeatedly traversed.

Worst-case complexity becomes very large.
