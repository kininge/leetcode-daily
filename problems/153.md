# 🧩 Problem #153 – Find Minimum in Rotated Sorted Array

**Difficulty:** Medium     
**Topics:** `Array`, `Binary Search`    
**Link:** [https://leetcode.com/problems/find-minimum-in-rotated-sorted-array/](https://leetcode.com/problems/find-minimum-in-rotated-sorted-array/)

---

## 📝 Problem Summary

You are given a sorted array that has been rotated between `1` and `n` times.

Example:

```txt
Original:
[0,1,2,4,5,6,7]
```

Possible rotations:

```txt
[4,5,6,7,0,1,2]
[0,1,2,4,5,6,7]
```

The array contains unique elements.

Your task is to return the minimum element in:

```txt
O(log n)
```

time.

---

### Key Observation

A rotated sorted array consists of two sorted parts:

```txt
[larger values] + [smaller values]
```

The minimum element is exactly the pivot point where rotation occurs.

---

### Constraints

* `1 <= nums.length <= 5000`
* `-5000 <= nums[i] <= 5000`
* All integers are unique
* Array is sorted and rotated

---

## 💡 Intuition

Since the array was originally sorted:

```txt
ascending order
```

rotation creates a structure where:

```txt
left side > right side
```

Example:

```txt
[4,5,6,7,0,1,2]
```

Here:

```txt
[4,5,6,7]
```

belongs to the first sorted portion.

And:

```txt
[0,1,2]
```

belongs to the second sorted portion.

The minimum element is the first element of the second portion.

This structure allows Binary Search.

---

## 🛠 Approach 1 — Linear Scan (Brute Force)

### Idea

Traverse the entire array and keep track of the minimum value.

---

### Code (TypeScript)

```ts
function findMin(nums: number[]): number {
    let min = 5000;

    for (const num of nums) {
        min = Math.min(min, num);
    }

    return min;
}
```

---

### Complexity

| Operation       | Complexity |
| --------------- | ---------- |
| Traversal       | `O(n)`     |
| Auxiliary Space | `O(1)`     |

---

## 🛠 Approach 2 — Binary Search (Pivot Detection)

### Idea

The minimum element is the pivot point where:

```txt
previous element > current element
```

or:

```txt
current element > next element
```

We use Binary Search to locate this pivot efficiently.

---

### Step 1 — Handle Edge Cases

If:

```txt
nums[0] < nums[n-1]
```

then the array is already sorted and not rotated.

So the first element is the minimum.

Also:

```txt
n === 1
```

trivially returns the only element.

Code:

```ts
if ((n === 1) || (nums[0] < nums[n - 1])) {
    return nums[0];
}
```

---

### Important Invariant

After the above check:

```txt
The array is guaranteed to be rotated.
```

So:

* minimum element cannot be at index `0`
* maximum element cannot be at index `n-1`

This guarantees:

```ts
nums[mid - 1]
nums[mid + 1]
```

remain valid during Binary Search.

---

### Step 2 — Detect Pivot

If:

```txt
nums[mid - 1] > nums[mid]
```

then:

```txt
nums[mid]
```

is the minimum.

Similarly:

```txt
nums[mid] > nums[mid + 1]
```

means:

```txt
nums[mid + 1]
```

is the minimum.

---

### Step 3 — Decide Search Direction

Observation:

Elements in the first rotated portion are always greater than:

```txt
nums[n - 1]
```

Example:

```txt
[4,5,6,7,0,1,2]
```

Here:

```txt
4,5,6,7 > 2
0,1,2 <= 2
```

So:

```ts
const isFirstPart = nums[mid] > nums[n - 1];
```

If:

```txt
true
```

search right side.

Otherwise:

```txt
search left side
```

---

### Why It Works

Binary Search continuously narrows the search space toward the rotation pivot.

Since the array contains unique values and consists of two sorted portions, the pivot can be identified in logarithmic time.

---

### Code (TypeScript)

```ts
function findMin(nums: number[]): number {
    const n = nums.length;

    // single element or already sorted
    if ((n === 1) || (nums[0] < nums[n - 1])) {
        return nums[0];
    }

    let left = 0;
    let right = n - 1;

    while (left <= right) {

        const mid = (left + right) >> 1;

        // current element is minimum
        if (nums[mid - 1] > nums[mid]) {
            return nums[mid];
        }

        // next element is minimum
        if (nums[mid] > nums[mid + 1]) {
            return nums[mid + 1];
        }

        const isFirstPart = nums[mid] > nums[n - 1];

        if (isFirstPart) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }
}
```

---

## 🛠 Approach 3 — Cleaner Binary Search

### Idea

Instead of explicitly detecting the pivot, we continuously eliminate the sorted half that cannot contain the minimum.

---

### Key Observation

If:

```txt
nums[mid] > nums[right]
```

then minimum must lie on the:

```txt
right side
```

because:

* left half is sorted
* pivot is not inside fully sorted portion

Otherwise:

```txt
minimum lies on left side including mid
```

---

### Why It Works

At every step:

* one half is guaranteed sorted
* the minimum cannot disappear
* search space reduces by half

Eventually:

```txt
left === right
```

which points to the minimum element.

---

### Code (TypeScript)

```ts
function findMin(nums: number[]): number {
    let left = 0;
    let right = nums.length - 1;

    while (left < right) {

        const mid = (left + right) >> 1;

        if (nums[mid] > nums[right]) {
            left = mid + 1;
        } else {
            right = mid;
        }
    }

    return nums[left];
}
```

| Linear Scan                     | `O(n)`          | `O(1)`          | Simple brute force               |
| Binary Search (Pivot Detection) | `O(log n)`      | `O(1)`          | Explicit pivot detection         |
| Cleaner Binary Search           | `O(log n)`      | `O(1)`          | Cleaner invariant-based solution |

---

## 🏆 Which Binary Search Is Better?

Both Binary Search solutions are correct.

However, the cleaner boundary-based version is usually preferred because:

* fewer conditions
* no neighbor checks
* easier invariants
* simpler debugging
* fewer edge-case risks

The pivot-detection approach is still valuable because it demonstrates strong understanding of rotated-array structure.

## 📌 Key Takeaways

* Rotated sorted arrays usually contain two sorted portions.

* Binary Search often works by identifying:

```txt
which half is guaranteed sorted
```

* Maintaining correct invariants is extremely important in Binary Search.

* Neighbor-check pivot detection is valid when edge cases guarantee pivot is not at boundaries.

* Cleaner Binary Search solutions often avoid explicit pivot detection entirely.

---

## ⚠️ Common Mistakes

### 1️⃣ Forgetting Already Sorted Case

If:

```txt
nums[0] < nums[n-1]
```

then the array is not rotated.

Minimum is simply:

```txt
nums[0]
```

---

### 2️⃣ Incorrect Mid Neighbor Access

Using:

```ts
nums[mid - 1]
nums[mid + 1]
```

without maintaining proper invariants may cause out-of-bound errors.

---

### 3️⃣ Wrong Search Direction

Incorrectly identifying which half contains the pivot can break Binary Search.

---

### 4️⃣ Using Linear Search Despite Logarithmic Requirement

The problem explicitly requires:

```txt
O(log n)
```

So brute-force traversal is not considered optimal.
