# 🧩 Problem #1665 – Minimum Initial Energy to Finish Tasks

**Difficulty:** Hard     
**Topics:** `Greedy`, `Sorting`, `Binary Search`, `Math`     
**Link:** [https://leetcode.com/problems/minimum-initial-energy-to-finish-tasks/](https://leetcode.com/problems/minimum-initial-energy-to-finish-tasks/)

---

## 📝 Problem Summary

You are given an array:

```txt
tasks[i] = [actuali, minimumi]
```

Where:

* `actuali` → energy spent after completing the task
* `minimumi` → minimum energy required before starting the task

You may complete tasks in any order.

Your goal is to find the **minimum initial energy** required to finish all tasks.

---

### Example

Task:

```txt
[10, 12]
```

Meaning:

* You need at least `12` energy to start
* After completion, energy decreases by `10`

If current energy is:

```txt
11 → cannot start
13 → can start → remaining energy = 3
```

---

### Key Observation

The order of tasks matters.

A bad ordering may require significantly higher starting energy.

So the real challenge is:

```txt
Find the optimal task ordering
```

---

### Constraints

* `1 <= tasks.length <= 10^5`
* `1 <= actuali <= minimumi <= 10^4`

---

## 💡 Intuition

At first glance, the problem may look like:

* Dynamic Programming
* Knapsack
* Binary Search on Answer

because:

* we need minimum possible energy
* multiple answers are possible
* tasks interact with remaining energy

However, the real solution is a Greedy ordering problem.

---

## ❌ Why Knapsack Does Not Apply

Knapsack problems usually involve:

```txt
Choose some items
```

But here:

```txt
Every task must be completed
```

So there is no:

* capacity optimization
* subset selection
* include/exclude decision

The only thing we control is:

```txt
Task ordering
```

---

## ❌ Binary Search Works — But Is Not Necessary

A natural observation is:

```txt
If X energy works,
then X + 1 also works.
```

This monotonic property makes binary search valid.

However:

* binary search still requires checking feasibility
* feasibility depends on ordering
* the real challenge is discovering the optimal order

Once the optimal ordering is known, we can compute the answer directly in one pass.

So binary search becomes unnecessary.

---

## 🛠 Approach 1 — Binary Search + Greedy Validation ()

### Idea

A natural observation is:

```txt
If X initial energy works,
then X + 1 will also work.
```

This monotonic property suggests:

```txt
Binary Search on Answer
```

For every guessed starting energy:

```txt
mid
```

we simulate task execution and check whether all tasks can be completed.

---

### Sorting Strategy

Tasks are sorted using:

```txt
minimum - actual
```

in descending order.

This attempts to prioritize tasks that require larger reserved energy.

---

### Feasibility Check

For every task:

* if current energy is at least `minimum`
* complete the task
* reduce energy by `actual`

Otherwise:

```txt
Current initial energy is insufficient.
```

---

### Code (TypeScript)

```ts
function minimumEffort(tasks: number[][]): number {

    tasks.sort(([act1, min1], [act2, min2]) => {
        const diff1 = min1 - act1;
        const diff2 = min2 - act2;

        return diff2 - diff1;
    });

    const isSatisfy = (n: number) => {
        let energy = n;

        for (const [actual, minimum] of tasks) {

            if (minimum <= energy) {
                energy -= actual;
            } else {
                return false;
            }
        }

        return true;
    };

    let minimumInitialEnergy = 1_000_000_000;

    let min = 0;
    let max = 1_000_000_000;

    while (min <= max) {

        const mid = (min + max) >> 1;

        if (isSatisfy(mid)) {

            minimumInitialEnergy = mid;

            // check further smaller solution
            max = mid - 1;

        } else {
            min = mid + 1;
        }
    }

    return minimumInitialEnergy;
}
```

### Complexity

A natural thought process for this problem is:

```txt
If X initial energy works,
then X + 1 will also work.
```

This creates a monotonic property:

```txt
smaller → invalid
larger → valid
```

which strongly suggests:

```txt
Binary Search on Answer
```

---

### Step 1 — Guess Initial Energy

We binary search the minimum possible starting energy.

For every guessed energy:

```txt
mid
```

we check whether all tasks can be completed.

---

### Step 2 — Try Greedy Ordering

A reasonable greedy intuition is:

```txt
Tasks with larger minimum energy
should be completed earlier.
```

So tasks are sorted by:

```txt
minimum descending
```

and if minimums are equal:

```txt
actual ascending
```

---

### Step 3 — Validate Feasibility

For every task:

* if current energy is at least `minimum`
* complete task
* decrease energy by `actual`

Otherwise:

```txt
mid is invalid
```

## ❌ Why This Approach Is Not Fully Optimal

Although the binary search idea is mathematically valid, the sorting strategy is not optimal.

Sorting only by:

```txt
minimum descending
```

fails to capture:

```txt
How restrictive the task actually is.
```

The true greedy property depends on:

```txt
minimum - actual
```

not just:

```txt
minimum
```

So while this approach works conceptually and demonstrates strong monotonicity recognition, it still misses the key greedy invariant.

| Binary Search     | `O(log RANGE)`   |
| Total Time        | `O(n log RANGE)` |
| Auxiliary Space   | `O(1)`           |

## 🔥 Core Greedy Insight

For every task:

```txt
[actual, minimum]
```

consider:

```txt
minimum - actual
```

This value represents:

```txt
How much extra reserved energy this task requires.
```

Tasks with larger reserve requirements are more restrictive.

Since energy only decreases over time, restrictive tasks should be completed earlier.

This leads to the greedy ordering:

```txt
Sort by:
(minimum - actual) descending
```

---

## 🧠 Mathematical Proof (Exchange Argument)

Suppose we have two tasks:

```txt
A = [a1, m1]
B = [a2, m2]
```

We compare two possible orders.

---

### Case 1 — Do A Before B

To start task A:

```txt
initial >= m1
```

After finishing A:

```txt
remaining = initial - a1
```

To start B:

```txt
initial - a1 >= m2
```

So:

```txt
initial >= a1 + m2
```

Therefore required energy becomes:

```txt
max(m1, a1 + m2)
```

---

### Case 2 — Do B Before A

Similarly:

```txt
max(m2, a2 + m1)
```

---

## 🔍 Which Order Is Better?

We prefer:

```txt
A before B
```

when:

```txt
max(m1, a1 + m2)
<=
max(m2, a2 + m1)
```

After simplifying using exchange argument analysis, the ordering condition becomes:

```txt
m1 - a1
>=
m2 - a2
```

Therefore:

```txt
Tasks with larger (minimum - actual)
should come first.
```

This proves the greedy ordering.

## 🛠 Approach 2 — Greedy Sorting (Optimal Solution)

### Step 1 — Sort Tasks

Sort tasks in descending order of:

```txt
minimum - actual
```

Code:

```ts
tasks.sort((a, b) => {
    return (b[1] - b[0]) - (a[1] - a[0]);
});
```

---

### Step 2 — Simulate Energy

Maintain:

* `answer` → minimum initial energy required
* `current` → current remaining energy

For each task:

If current energy is insufficient:

```txt
current < minimum
```

we must increase initial energy.

Required increment:

```txt
minimum - current
```

Then complete the task:

```txt
current -= actual
```

---

## ✅ Why This Works

Sorting by:

```txt
minimum - actual
```

ensures:

* highly restrictive tasks are completed earlier
* later tasks become easier because they require smaller reserve energy

The greedy order minimizes the maximum energy requirement encountered during execution.

## 💻 Code (TypeScript)

```ts
function minimumEffort(tasks: number[][]): number {

    tasks.sort((a, b) => {
        return (b[1] - b[0]) - (a[1] - a[0]);
    });

    let answer = 0;
    let current = 0;

    for (const [actual, minimum] of tasks) {

        if (current < minimum) {
            answer += (minimum - current);
            current = minimum;
        }

        current -= actual;
    }

    return answer;
}
```

## 📌 Key Takeaways

* Hard greedy problems often depend on discovering the correct ordering.
* Binary search may work mathematically but still miss the real optimization insight.
* Exchange arguments are common in sorting-based greedy proofs.
* The quantity:

```txt
minimum - actual
```

represents reserve energy pressure.

* Tasks with larger reserve requirements should be completed earlier.

---

## ⚠️ Common Mistakes

### 1️⃣ Sorting By Minimum Descending

Sorting only by:

```txt
minimum descending
```

does not correctly capture reserve energy pressure.

The real greedy metric is:

```txt
minimum - actual
```

---

### 2️⃣ Assuming Order Does Not Matter

Task ordering completely changes the required initial energy.

This is the central challenge of the problem.

---

### 3️⃣ Overcomplicating With DP

Although the problem looks like DP initially, there are no overlapping state transitions worth memoizing.

The optimal solution comes from greedy ordering.

---

### 4️⃣ Using Binary Search Without Optimal Ordering

Binary search can verify whether an answer works.

But without the correct task ordering, feasibility checking itself becomes unreliable.

The greedy insight is still necessary.
