# 🧩 Problem #154 – Find Minimum in Rotated Sorted Array II

**Difficulty:** Hard    
**Topics:** `Array`, `Binary Search`     
**Link:** [https://leetcode.com/problems/find-minimum-in-rotated-sorted-array-ii/](https://leetcode.com/problems/find-minimum-in-rotated-sorted-array-ii/)

---

## 📝 Problem Summary

You are given a sorted array that has been rotated between `1` and `n` times.

Unlike Problem 153, this array may contain duplicate values.

Your task is to return the minimum element while minimizing the number of operations.

---

### Example

```txt
[4,5,6,7,0,1,2]
→ minimum = 0
```

With duplicates:

```txt
[2,2,2,0,1]
→ minimum = 0
```

---

### Key Observation

This problem is almost identical to:

```txt
153. Find Minimum in Rotated Sorted Array
```

However:

```txt
Duplicates break some Binary Search guarantees.
```

That subtle ambiguity is the core difficulty.

---

### Constraints

* `1 <= nums.length <= 5000`
* `-5000 <= nums[i] <= 5000`
* Array is sorted and rotated
* Duplicates are allowed

---

## 💡 Intuition

In Problem 153 (unique elements), we could reliably determine:

```txt
Which side contains the pivot
```

using:

```txt
nums[mid] > nums[right]
```

or:

```txt
nums[mid] < nums[right]
```

because all values were unique.

---

## ❌ Why Duplicates Make This Hard

Consider:

```txt
[2,2,2,0,1]
```

Easy:

```txt
nums[mid] > nums[right]
```

still gives useful information.

But now consider:

```txt
[1,1,1,0,1]
```

or:

```txt
[1,1,1,1,1]
```

When:

```txt
nums[mid] == nums[right]
```

we lose information.

We can no longer determine:

* which side contains the minimum
* whether left or right portion is sorted
* where pivot exists

This ambiguity weakens Binary Search.

---

## 🛠 Approach 1 — Remove Duplicates First

### Idea

One possible idea is:

1. Remove duplicate values
2. Reduce problem into Problem 153
3. Apply standard Binary Search pivot detection

---

### Why It Works

Removing duplicates restores the unique-element property.

Then we can reuse the original rotated-array Binary Search logic.

---

### Drawback

Removing duplicates itself requires:

```txt
O(n)
```

traversal and additional storage.

So overall complexity becomes:

```txt
O(n)
```

which loses the main advantage of Binary Search.

---

### Code (TypeScript)

```ts
function findMin(nums: number[]): number {
    let n = nums.length;

    if (n === 1) {
        return nums[0];
    }

    const store = new Set<number>();

    const uniqueNums: number[] = [];

    for (const num of nums) {

        if (!store.has(num)) {
            store.add(num);
            uniqueNums.push(num);
        }
    }

    n = uniqueNums.length;

    if ((n === 1) || (uniqueNums[0] < uniqueNums[n - 1])) {
        return uniqueNums[0];
    }

    let left = 0;
    let right = n - 1;

    while (left <= right) {

        const mid = (left + right) >> 1;

        if (uniqueNums[mid - 1] > uniqueNums[mid]) {
            return uniqueNums[mid];
        }

        const isInFirstHalf = uniqueNums[mid] > uniqueNums[n - 1];

        if (isInFirstHalf) {
            left = mid + 1;
        }
        else {
            right = mid - 1;
        }
    }
}
```

## 🛠 Approach 2 — Optimal Binary Search With Duplicates

### Core Idea

We still use Binary Search.

But now we must carefully handle ambiguity introduced by duplicates.

---

### Case 1 — Minimum On Right Side

If:

```txt
nums[mid] > nums[right]
```

then:

```txt
minimum lies on the right side
```

because:

* left half remains sorted
* pivot cannot exist there

So:

```ts
left = mid + 1
```

---

### Case 2 — Minimum On Left Side

If:

```txt
nums[mid] < nums[right]
```

then:

```txt
minimum lies on left side including mid
```

So:

```ts
right = mid
```

---

### Case 3 — Duplicate Ambiguity (Hard Part)

If:

```txt
nums[mid] == nums[right]
```

we lose directional information.

Example:

```txt
[1,1,1,0,1]
```

We cannot determine:

* whether pivot is left
* whether pivot is right

because duplicates hide ordering information.

---

### Safe Reduction Trick

When:

```txt
nums[mid] == nums[right]
```

we safely shrink search space:

```ts
right--
```

Why is this safe?

Because:

```txt
nums[right] == nums[mid]
```

Removing one duplicate does not remove unique minimum information.

---

## 🔥 Why Worst Case Becomes O(n)

In ideal Binary Search:

```txt
search space halves every iteration
```

But with duplicates:

```txt
right--
```

may reduce search space by only:

```txt
1 element
```

Example:

```txt
[1,1,1,1,1,1]
```

Every iteration becomes:

```txt
nums[mid] == nums[right]
```

So worst-case complexity degrades to:

```txt
O(n)
```

This is the key follow-up insight of the problem.

---

### Why It Still Works Well

Although worst case is:

```txt
O(n)
```

most practical cases still behave close to:

```txt
O(log n)
```

because ambiguity does not occur continuously.

---

### Code (TypeScript)

```ts
function findMin(nums: number[]): number {

    let left = 0;
    let right = nums.length - 1;

    while (left < right) {

        const mid = (left + right) >> 1;

        if (nums[mid] > nums[right]) {

            left = mid + 1;

        }
        else if (nums[mid] < nums[right]) {

            right = mid;

        }
        else {

            right--;
        }
    }

    return nums[left];
}
```

| Remove Duplicates First       | `O(n)`                            | `O(n)`          | Simplifies problem but loses logarithmic advantage |
| Binary Search With Duplicates | Average: `O(log n)` Worst: `O(n)` | `O(1)`          | Optimal and interview-preferred                    |

### Optimal Binary Search

| Case            | Complexity |
| --------------- | ---------- |

| Worst Case      | `O(n)`     |
| Auxiliary Space | `O(1)`     |

## 📌 Key Takeaways

* Duplicates weaken Binary Search guarantees.

* The difficult part is handling:

```txt
nums[mid] == nums[right]
```

* Binary Search sometimes degrades from:

```txt
O(log n)
→
O(n)
```

when duplicates destroy directional information.

* Shrinking search space safely is often enough even when exact direction cannot be determined.

---

## ⚠️ Common Mistakes

### 1️⃣ Assuming Duplicates Do Not Affect Binary Search

Duplicates may completely hide pivot direction.

---

### 2️⃣ Forgetting Worst Case Degradation

Binary Search no longer guarantees:

```txt
O(log n)
```

in every case.

---

### 3️⃣ Incorrectly Discarding Both Sides

When:

```txt
nums[mid] == nums[right]
```

we cannot confidently discard half the array.

Only safe reduction is:

```txt
right--
```

---

### 4️⃣ Overcomplicating With Duplicate Removal

Removing duplicates works logically but sacrifices efficiency and additional space.
