# 🧩 Problem #2685 - Count the Number of Complete Components

**Difficulty:** Medium    
**Topics:** `Union Find`, `Graph`       
**Link:** https://leetcode.com/problems/count-the-number-of-complete-components/description   

---

## 💡 Intuition

The problem asks us to count **complete connected components**.

There are two separate tasks:

1. Find all connected components.
2. Verify whether each component is complete.

A **complete graph** with `k` nodes has the property that every node is connected to every other node.

Therefore, every node inside the component must have

```text
degree = k - 1
```

Since connectivity and completeness are independent concerns, we can solve them separately.

* **Union-Find** identifies connected components.
* **Degree information** determines whether a component is complete.

## 🔍 Key Observation

Finding connected components is **not enough**.

For example,

```text
0
| \
1  2
```

This graph is connected.

However,

```text
1 --- 2
```

is missing.

So although all nodes belong to the same component, it is **not** complete.

The missing step after Union-Find is verifying that every node satisfies

```text
degree = componentSize - 1
```

---

# 🧠 Approach 1

### Idea

After building Union-Find,

collect every node belonging to each component.

Example

```text
Root

↓

[0,1,2]
```

Then verify

```text
For every node

degree == componentSize - 1
```

If every node satisfies the condition,

the component is complete.

### Solution

```ts
var countCompleteComponents = function(n, edges) {
    const graph = new Map();
    const disjointSet = new DisjointSet(n);

    for (const [node1, node2] of edges) {
        disjointSet.union(node1, node2);

        graph.set(node1, (graph.get(node1) ?? 0) + 1);
        graph.set(node2, (graph.get(node2) ?? 0) + 1);
    }

    const components = disjointSet.getAllComponents();

    let completeComponentsCount = 0;

    components.forEach((nodes) => {
        if (nodes.length === 1) {
            completeComponentsCount++;
            return;
        }

        const requiredDegree = nodes.length - 1;

        let complete = true;

        for (const node of nodes) {
            if ((graph.get(node) ?? 0) < requiredDegree) {
                complete = false;
                break;
            }
        }

        if (complete) {
            completeComponentsCount++;
        }
    });

    return completeComponentsCount;
};
```

---

### Observation

This solution is straightforward and easy to reason about because each component is verified independently.

The downside is that every component explicitly stores all of its nodes.

---

# 🧠 Approach 2

### Improvement

Instead of storing

```text
component

↓

[node1,node2,node3...]
```

we observe that Union-Find already stores

```text
component size
```

So we only need

* component size
* degree of each node

During one pass over all nodes:

* Optimistically assume a component is complete.
* If any node violates

```text
degree == componentSize - 1
```

invalidate that component.

This removes the need to explicitly group nodes by component.

## ✅ Final Solution

```ts
function countCompleteComponents(n: number, edges: number[][]): number {
    const unionFind = new MyUnionFind(n);
    const graph = new Map();

    for (const [u, v] of edges) {
        unionFind.union(u, v);

        let edgesU = [];
        if (graph.has(u)) edgesU = graph.get(u);
        edgesU.push(v);
        graph.set(u, edgesU);

        let edgesV = [];
        if (graph.has(v)) edgesV = graph.get(v);
        edgesV.push(u);
        graph.set(v, edgesV);
    }

    let completeConnectedComponents = 0;

    const completedComponets = new Set();
    const nonCompletedComponets = new Set();

    for (let i = 0; i < n; i++) {
        const rootNode = unionFind.find(i);

        if (nonCompletedComponets.has(rootNode)) continue;

        const nodes = unionFind.componentSize(rootNode);
        const degree = graph.has(i) ? graph.get(i).length : 0;

        if (degree === nodes - 1) {
            if (!completedComponets.has(rootNode)) {
                completeConnectedComponents++;
                completedComponets.add(rootNode);
            }
        } else {
            if (completedComponets.has(rootNode)) {
                completeConnectedComponents--;
            }

            nonCompletedComponets.add(rootNode);
        }
    }

    return completeConnectedComponents;
}
```

## ⚖️ Why it Works

Union-Find guarantees that every node belongs to exactly one connected component.

For each node we know

* its component size
* its degree

If every node inside the component satisfies

```text
degree = componentSize - 1
```

then every node is connected to every other node.

Therefore the component is complete.

If any node violates this property, the component is immediately marked as invalid and ignored for the remaining nodes.

## 🔄 Comparison of Both Approaches

| Aspect            | Approach 1 (22 Mar 2025)             | Approach 2 (11 Jul 2026 )               |
| ----------------- | ------------------------------------ | --------------------------------------- |
| Component Storage | Stores every node in every component | Uses Union-Find component size directly |
| Verification      | Iterate through stored nodes         | Verify while scanning all nodes         |
| Extra Memory      | `O(n)` for component lists           | Only two sets of component roots        |
| Time Complexity   | `O(E · α(n) + n)`                    | `O(E · α(n) + n)`                       |
| Readability       | Easier to understand                 | Slightly more clever                    |
| Scalability       | Good                                 | Better memory usage                     |

### Why the Final Solution is Better

The first solution stores

```text
root

↓

[node1,node2,node3]
```

even though Union-Find already knows the component size.

The final solution avoids storing component members entirely.

Instead, it verifies completeness while scanning the nodes once.

This makes the solution more memory efficient without increasing time complexity.

## 🏆 Key Takeaways

* Separate **finding connected components** from **verifying properties** of those components.
* Union-Find is responsible only for connectivity.
* Component completeness can be verified using node degrees.
* Reusing information already maintained by Union-Find (component size) can eliminate unnecessary data structures.

## 📚 Pattern Learned

### Component Verification Pattern

Whenever a problem asks

* Count connected components satisfying a property
* Verify every node inside a component
* Check if a component forms a tree, clique, or other structure

think

```text
Union Find / DFS

↓

Identify components

↓

Verify the required property
inside each component
```

Keep the two phases conceptually separate.
