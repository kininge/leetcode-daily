# 🧩 Problem #33 – Search in Rotated Sorted Array

**Difficulty:** Medium   
**Topics:** `Array`, `Binary Search`   
**Link:** [https://leetcode.com/problems/search-in-rotated-sorted-array/](https://leetcode.com/problems/search-in-rotated-sorted-array/)   

---

## 📝 Problem Summary

You are given a sorted array that may have been rotated at some pivot.

Example:

```txt
Original:
[0,1,2,4,5,6,7]
```

Possible rotation:

```txt
[4,5,6,7,0,1,2]
```

The array contains unique values.

Your task is to find the index of a target value in:

```txt
O(log n)
```

time.

If target does not exist:

```txt
return -1
```

---

### Constraints

* `1 <= nums.length <= 5000`
* `-10^4 <= nums[i] <= 10^4`
* All values are unique
* Array is sorted and possibly rotated

---

## 💡 Intuition

A rotated sorted array is actually composed of:

```txt
TWO sorted partitions
```

Example:

```txt
[4,5,6,7,0,1,2]
```

Partitions:

```txt
[4,5,6,7]
[0,1,2]
```

The challenge is determining:

```txt
Which side can still contain the target?
```

This leads naturally to Binary Search.

---

## 🔥 Important Observation

In rotated sorted arrays:

```txt
Elements in first partition
>
last element
```

Example:

```txt
[4,5,6,7,0,1,2]
```

Here:

```txt
4,5,6,7 > 2
0,1,2 <= 2
```

So:

```txt
nums[n-1]
```

acts like a partition boundary.

This becomes the foundation of the partition-classification solution.

---

## 🛠 Approach 1 — Partition Classification Binary Search

### Core Idea

Instead of explicitly detecting:

```txt
which half is sorted
```

we classify values into:

```txt
first partition
second partition
```

using:

```ts
nums[x] > nums[n-1]
```

If:

```txt
target and mid belong to same partition
```

then normal Binary Search rules apply.

Otherwise:

```txt
search opposite partition
```

---

### Step 1 — Identify Target Partition

If:

```txt
target > nums[n-1]
```

then target belongs to:

```txt
first partition
```

Otherwise:

```txt
second partition
```

---

### Step 2 — Identify Mid Partition

Similarly:

```ts
nums[mid] > nums[n-1]
```

determines where `mid` belongs.

---

### Step 3 — Same Partition Logic

If:

```txt
target and mid are in same partition
```

that partition remains sorted.

So:

```txt
normal binary search comparisons work
```

---

### Step 4 — Different Partition Logic

If:

```txt
target and mid belong to different partitions
```

one side becomes impossible.

So we move toward the correct partition.

---

### Why It Works

The rotation creates two sorted regions.

Using the last element as a partition separator allows us to identify where each value belongs.

Binary Search then narrows search space correctly while preserving:

```txt
O(log n)
```

complexity.

---

### Code (TypeScript)

```ts
function search(nums: number[], target: number): number {

    const n = nums.length;

    let left = 0;
    let right = n - 1;

    const isTargetInFirstPart =
        target > nums[n - 1];

    while (left <= right) {

        const mid = (left + right) >> 1;

        if (nums[mid] === target) {
            return mid;
        }

        const isMidInFirstPart =
            nums[mid] > nums[n - 1];

        const samePartition =
            isTargetInFirstPart === isMidInFirstPart;

        // normal binary search
        if (samePartition) {

            if (target < nums[mid]) {
                right = mid - 1;
            }
            else {
                left = mid + 1;
            }
        }
        // move toward correct partition
        else {

            if (isMidInFirstPart) {
                left = mid + 1;
            }
            else {
                right = mid - 1;
            }
        }
    }

    return -1;
}
```

## 🛠 Approach 2 — Sorted Half Detection (Classic)

### Core Idea

At every step:

```txt
one half is guaranteed sorted
```

We identify the sorted half and determine whether target lies inside it.

If target belongs there:

```txt
search inside sorted half
```

Otherwise:

```txt
search opposite half
```

---

### Step 1 — Detect Sorted Half

If:

```txt
nums[left] <= nums[mid]
```

then:

```txt
left half is sorted
```

Otherwise:

```txt
right half is sorted
```

---

### Step 2 — Decide Search Direction

If target lies inside sorted range:

```txt
continue search there
```

Otherwise:

```txt
search opposite side
```

---

### Why It Works

Even after rotation:

```txt
at least one side always remains sorted
```

This property allows Binary Search to safely eliminate half the search space each iteration.

---

### Code (TypeScript)

```ts
function search(nums: number[], target: number): number {

    let left = 0;
    let right = nums.length - 1;

    while (left <= right) {

        const mid = (left + right) >> 1;

        if (nums[mid] === target) {
            return mid;
        }

        // left half sorted
        if (nums[left] <= nums[mid]) {

            if (
                nums[left] <= target &&
                target < nums[mid]
            ) {
                right = mid - 1;
            }
            else {
                left = mid + 1;
            }
        }
        // right half sorted
        else {

            if (
                nums[mid] < target &&
                target <= nums[right]
            ) {
                left = mid + 1;
            }
            else {
                right = mid - 1;
            }
        }
    }

    return -1;
}
```

## 🏆 Which Approach Is Better?

Both approaches are:

```txt
O(log n)
```

and fully optimal.

---

### Partition Classification

Pros:

* elegant invariant
* mathematically clean
* avoids repeated sorted-half checks

Cons:

* less intuitive initially
* harder to derive during interviews

---

### Sorted Half Detection

Pros:

* more intuitive
* easier to explain
* very commonly recognized

Cons:

* more condition-heavy
* slightly noisier logic

## 📌 Key Takeaways

* Rotated sorted arrays consist of two sorted partitions.

* Binary Search works by eliminating impossible halves.

* Multiple valid Binary Search invariants may exist for the same problem.

* Using:

```txt
nums[n-1]
```

as a partition separator is an elegant technique.

* Confusion in rotated-array problems is common because two concepts overlap:

```txt
sorted ordering
+
rotation partitioning
```

---

## ⚠️ Common Mistakes

### 1️⃣ Forgetting Rotation Creates Two Sorted Regions

The array is not fully unsorted.

It still preserves partial ordering.

---

### 2️⃣ Incorrect Partition Classification

Partition logic relies on:

```txt
unique values
```

Duplicates break this approach.

---

### 3️⃣ Mixing Sorted-Half Logic And Partition Logic

Combining two mental models incorrectly often creates bugs.

---

### 4️⃣ Infinite Binary Search Loops

Incorrect pointer updates may prevent search space reduction.

Always ensure:

```txt
left/right strictly move
```
