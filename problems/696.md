# 🧩 Problem #696 – Count Binary Substrings

**Difficulty:** Easy  
**Topics:**  `String`, `Two Pointers`  
**Link:** https://leetcode.com/problems/count-binary-substrings/

---

## 📝 Problem Summary

Given a binary string `s`, return the number of non-empty substrings that:

1. Contain equal number of `0`s and `1`s  
2. Have all `0`s grouped together and all `1`s grouped together  

Important clarifications:

- Substrings must be contiguous.
- Valid substrings look like:
  - `"01"`, `"10"`
  - `"0011"`, `"1100"`
  - `"000111"`, etc.
- Patterns like `"0101"` are **invalid** because digits are not grouped consecutively.

### Example

Input: `s = "00110011"`
Valid substrings: `"0011", "01", "1100", "10", "0011", "01"`
Output: `6`


### Constraints

- `1 <= s.length <= 10^5`
- `s[i]` is either `'0'` or `'1'`

---

The key observation:
Valid substrings always occur at **boundaries between consecutive groups** of 0s and 1s.

---

## 💡 Intuition

A valid substring must look like:

```txt
000111
111000
01
10
```

Notice something important:

- All valid substrings are formed at the **boundary between two consecutive groups**.
- If a group of `0`s is followed by a group of `1`s (or vice versa),
  the number of valid substrings formed between them equals: `min(size_of_left_group, size_of_right_group)`

### Why does this work?

Example: `000111`

- Left group size = 3
- Right group size = 3

Possible valid substrings:

```txt
01
0011
000111
```

Count = `min(3, 3) = 3`

Another example: `00111`

- Left group size = 2
- Right group size = 3

Valid substrings:

```txt
01
0011
```

Count = `min(2, 3) = 2`

---

### Core Insight

Instead of checking all substrings (which would be O(n²)):

1. Break the string into consecutive groups.
2. For every adjacent pair of groups:
   - Add `min(prevGroupLength, currGroupLength)` to the answer.

This reduces the problem to a **single pass grouping problem**, enabling an O(n) solution.

---

## 🛠 Approach 1 — Simulation with Separate Counters (Basic / Less Clean)

### Idea

Traverse the string while maintaining:

- `zeroCount` → length of current consecutive `0` group
- `oneCount` → length of current consecutive `1` group

At every character:

- Extend the current group
- Reset the opposite counter when group changes
- If previous group length ≥ current group length,
  increment the substring count

This works because each time the current group grows,
a valid substring is formed as long as the previous group
is large enough to match it.

---

### Why it works

A valid substring must consist of: 
`[ some 0s ][ some 1s ]`
OR
`[ some 1s ][ some 0s ]`

If the previous group has at least as many characters
as the current growing group, a new valid substring is formed.

So at each step:

if previousGroupLength >= currentGroupLength
→ count++



This logic is implemented indirectly using two counters.

---

### Code (TypeScript)

```ts
function countBinarySubstrings(s: string): number {
    let oneCount = 0;
    let zeroCount = 0;
    let substringCount = 0;

    for (let i = 0; i < s.length; i++) {
        if (s[i] === '1') {
            if (i > 0 && s[i - 1] === '1') {
                oneCount++;
            } else {
                oneCount = 1;
            }

            if (zeroCount >= oneCount) {
                substringCount++;
            }
        } else {
            if (i > 0 && s[i - 1] === '0') {
                zeroCount++;
            } else {
                zeroCount = 1;
            }

            if (oneCount >= zeroCount) {
                substringCount++;
            }
        }
    }

    return substringCount;
}
```

### Time Complexity: 
- `O(n)` Single pass through the string.

### Space Complexity
- `O(1)` Only counters are used.

### When to use this approach

- When deriving logic incrementally during an interview.
- When you intuitively think in terms of tracking 0s and 1s.
- When you haven’t yet formalized the grouping insight.

### Limitations

- Logic is slightly harder to reason about.
- Counters implicitly represent previous/current groups.
- Less elegant compared to grouping-based formulation.
- Not as clean for explanation under interview pressure.

## 🚀 Approach 2 — Run-Length Grouping with `min(prev, curr)` (Improved & Cleaner)

### Idea

Instead of tracking `0` and `1` separately, we track:

- `prevRun` → length of previous consecutive group
- `currRun` → length of current consecutive group

When the character changes:
- Add `min(prevRun, currRun)` to the answer.
- Shift `currRun` into `prevRun`.
- Reset `currRun`.

Finally, add one last `min(prevRun, currRun)` after the loop.

---

### Why it works

For every adjacent pair of groups: `000111`


- `prevRun = 3`
- `currRun = 3`

Valid substrings formed = `min(3,3) = 3`

This works because:

Each valid substring must take:
- `k` characters from left group
- `k` characters from right group

Where: 
```txt
1 ≤ k ≤ min(prevRun, currRun)
```


So total substrings at that boundary = `min(prevRun, currRun)`.

This directly models the mathematical structure of the problem.

---

### Code (TypeScript)

```ts
function countBinarySubstrings(s: string): number {
    let prevRun = 0;
    let currRun = 1;
    let result = 0;

    for (let i = 1; i < s.length; i++) {
        if (s[i] === s[i - 1]) {
            currRun++;
        } else {
            result += Math.min(prevRun, currRun);
            prevRun = currRun;
            currRun = 1;
        }
    }

    result += Math.min(prevRun, currRun);

    return result;
}
```

### Time Complexity
- `O(n)` Single pass through the string.

### Space Complexity
- `O(1)` Only two counters are maintained.

### Why this is better than Approach 1

- Explicitly models the grouping structure.
- Cleaner reasoning.
- Easier to explain in interviews.
- Avoids separate tracking of `0` and `1`.
- Directly derived from mathematical observation.

This is the preferred interview solution.

## 📊 Comparison of Approaches

| Criteria | 🛠 Approach 1 — Dual Counters Simulation | 🚀 Approach 2 — Run-Length Grouping |
|-----------|-------------------------------------------|-------------------------------------|
| Core Idea | Track `zeroCount` and `oneCount` separately and increment when previous ≥ current | Track `prevRun` and `currRun` and add `min(prevRun, currRun)` |
| Abstraction Level | Character-based reasoning | Group-based reasoning |
| Mathematical Clarity | Implicit | Explicit |
| Code Readability | Slightly harder to reason about | Cleaner and more structured |
| Risk of Logical Bugs | Higher (conditional symmetry handling) | Lower (uniform logic) |
| Time Complexity | O(n) | O(n) |
| Space Complexity | O(1) | O(1) |
| Interview Explainability | Moderate | Strong |
| Production Readability | Acceptable | Preferred |

---

### 🏆 Final Recommendation

Both approaches are **optimal in time and space**, but:

- Approach 1 works, but reasoning is indirect.
- Approach 2 models the problem structure directly and is easier to justify.

For interviews and clean production code,  
**Approach 2 is the recommended solution.**
