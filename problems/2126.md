# 🧩 Problem #2126 – Destroying Asteroids

**Difficulty:** Medium   
**Topics:** `Greedy`, `Sorting`, `Heap`   
**Link:** https://leetcode.com/problems/destroying-asteroids/

---

# 📖 Problem Summary

We have a planet with initial mass:

```txt
mass
```

If:

```txt
planetMass >= asteroidMass
```

the asteroid is destroyed and:

```txt
planetMass += asteroidMass
```

Otherwise the planet is destroyed.

We may choose the asteroid collision order arbitrarily.

Return whether all asteroids can be destroyed.

---

# 💡 Intuition

The important statement is:

```txt
You may choose any order.
```

Initially this feels like:

```txt
Find the best asteroid to destroy next.
```

which naturally leads to:

```txt
Heap
```

because the set of destroyable asteroids changes as the planet grows.

But after thinking further, the problem turns out to have a much simpler greedy property.

---

# 🔍 Key Observation

After sorting:

```txt
a1 <= a2 <= a3 <= ...
```

Suppose:

```txt
planetMass < a1
```

Then:

```txt
planetMass < a2
planetMass < a3
...
```

as well.

Meaning:

```txt
No asteroid can be destroyed.
```

This gives a powerful conclusion:

```txt
If the smallest remaining asteroid
cannot be destroyed,

then no valid ordering exists.
```

---

# 🛠️ Approach 1 — Heap

### Idea

Continuously maintain:

```txt
Destroyable asteroids
```

and

```txt
Currently too large asteroids
```

using two heaps.

---

### Heap Meaning

#### Max Heap

Stores:

```txt
asteroidMass <= currentPlanetMass
```

These asteroids can be destroyed immediately.

---

#### Min Heap

Stores:

```txt
asteroidMass > currentPlanetMass
```

These asteroids are currently unavailable.

---

### Process

Destroy the largest available asteroid:

```txt
Max Heap
    ↓
Increase planet mass
    ↓
Move newly available asteroids
from Min Heap → Max Heap
```

Repeat until:

* all asteroids destroyed
* or no asteroid can be destroyed

---

### Why It Works

Every time planet mass increases:

```txt
Some previously unavailable asteroids
may become available.
```

The heaps efficiently maintain this partition.

---

# 🛠️ Approach 2 — Greedy (Optimal)

### Observation

If the smallest asteroid cannot be destroyed:

```txt
Nothing can be destroyed.
```

Therefore:

```txt
Destroying the smallest asteroid first
is always safe.
```

---

### Process

Sort asteroids.

Process left to right:

```txt
smallest
    ↓
larger
    ↓
larger
```

If any asteroid is larger than current mass:

```txt
return false
```

Otherwise:

```txt
planetMass += asteroid
```

---

# 🧪 Example

```txt
mass = 10

asteroids = [3,9,19,5,21]
```

Sort:

```txt
[3,5,9,19,21]
```

Planet growth:

```txt
10
↓
13
↓
18
↓
27
↓
46
↓
67
```

All asteroids destroyed.

Answer:

```txt
true
```

---

# ✅ Why Greedy Works

Suppose sorted order fails at:

```txt
asteroid[i]
```

Meaning:

```txt
planetMass < asteroid[i]
```

Since:

```txt
asteroid[i]
```

is the smallest remaining asteroid,

every remaining asteroid satisfies:

```txt
remaining >= asteroid[i]
```

Therefore:

```txt
planetMass
<
all remaining asteroids
```

No valid move exists.

Thus:

```txt
If greedy fails,
every possible ordering fails.
```

which proves correctness.

# 💻 Heap Code

```ts
function asteroidsDestroyed(mass: number, asteroids: number[]): boolean {
    const minHeapComparator = (childNode, parentNode) => {
        return childNode < parentNode;
    }

    const maxHeapComparator = (childNode, parentNode) => {
        return childNode > parentNode;
    }

    const minHeap = new CHeap(minHeapComparator);
    const maxHeap = new CHeap(maxHeapComparator);

    for (const asteroidMass of asteroids) {
        if (asteroidMass <= mass) {
            maxHeap.append(asteroidMass);
        } else {
            minHeap.append(asteroidMass);
        }
    }

    let planetMass = mass;

    function balanceOut(): void {
        while (
            minHeap.peak() !== null &&
            minHeap.peak() <= planetMass
        ) {
            const asteroid = minHeap.pop();
            maxHeap.append(asteroid);
        }
    }

    while (
        maxHeap.peak() !== null &&
        maxHeap.peak() <= planetMass
    ) {
        const asteroid = maxHeap.pop();

        planetMass += asteroid;

        balanceOut();

        if (minHeap.peak() === null) {
            return true;
        }
    }

    return false;
}
```

# 💻 Greedy Code

```ts
function asteroidsDestroyed(
    mass: number,
    asteroids: number[]
): boolean {

    asteroids.sort((a, b) => a - b);

    let planetMass = mass;

    for (const asteroid of asteroids) {

        if (asteroid > planetMass) {
            return false;
        }

        planetMass += asteroid;
    }

    return true;
}
```

# 🎯 Pattern Recognition

### Pattern

```txt
Greedy + Sorting
```

---

### Signal

Whenever a problem says:

```txt
You may reorder elements arbitrarily
```

consider:

```txt
Sort first
```

before reaching for:

* Heap
* Priority Queue
* Dynamic candidate selection

---

### Core Reduction

```txt
Can destroy asteroid[i]
        ↓
Can destroy all smaller asteroids
        ↓
Process smallest first
```

# 🧠 Key Takeaways

✅ Reordering freedom often suggests greedy.

✅ Before using a heap, check whether sorting alone is enough.

✅ If the smallest remaining option is impossible, the entire problem may be impossible.

✅ Sometimes a valid heap solution hides a stronger greedy observation.

---

# ⚠️ Common Mistakes

### ❌ Assuming larger-first is better

Destroying a larger asteroid first is unnecessary.

The smallest asteroid already gives the strongest proof.

---

### ❌ Jumping directly to Heap

Heap works, but solving candidate management is unnecessary once the greedy property is discovered.

---

### ❌ Missing the Greedy Proof

The solution is not:

```txt
Sort because it works.
```

The solution is:

```txt
Sort because failure of the smallest asteroid
implies failure of every possible ordering.
```
