# 🧩 Problem #868 – Binary Gap

**Difficulty:** Easy  
**Topics:** `Bit Manipulation`, `Math`  
**Link:** https://leetcode.com/problems/binary-gap/

---

## 📝 Problem Summary

Given a positive integer `n`, return the longest distance between two adjacent `1`s in its binary representation.

Two `1`s are considered adjacent if there are only `0`s separating them (possibly none).

The **distance** between two `1`s** is defined as the absolute difference between their bit positions.

If there are fewer than two `1`s in the binary representation, return `0`.

---

### Example

Input: `n = 9`

Binary representation: `1001`
Output: 3

 
Explanation:

The two `1`s are at positions `0` and `3`.  
Distance = `3 - 0 = 3`.

---

### Constraints

- `1 <= n <= 10^9`

---

## 🔎 Key Observation

We only care about:

- Positions of `1` bits in the binary representation.
- Distance between consecutive `1`s.

We do **not** need to:
- Generate substrings
- Store the entire binary representation
- Use extra data structures

Since `1 <= n <= 10^9`, the number fits within **30 bits**.

So the problem reduces to:

1. Traverse through the bits of `n`.
2. Whenever a `1` is found:
   - Compare its position with the previous `1`.
   - Update the maximum distance.
3. If fewer than two `1`s exist → return `0`.

---

### Core Insight

This is fundamentally a **bit position tracking problem**, not a string problem.

We only need to remember:

- The last index where a `1` was seen.
- The maximum distance found so far.

---

## 🛠 Approach 1 — Fixed 31-Bit Scan (Bit Masking)

### 💡 Idea

Loop through bit positions from `0` to `30`:

- Use bit masking to check whether the `i`-th bit is set.
- Track the index of the previous `1`.
- Update the maximum distance whenever a new `1` is found.

Since `n <= 10^9`, it fits within 30 bits.  
So iterating up to 31 positions is safe.

---

### Code (TypeScript)

```ts
function binaryGap(n: number): number {
    let distance = 0;
    let lastOneIndex = -1;

    for (let i = 0; i < 31; i++) {
        if ((n & (1 << i)) !== 0) {
            if (lastOneIndex !== -1) {
                distance = Math.max(distance, i - lastOneIndex);
            }
            lastOneIndex = i;
        } 
    }

    return distance;
}
```

---

### Why It Works

- `(1 << i)` creates a mask where only the `i-th` bit is set.
- `n & mask` checks whether that bit is set in `n`.
- Each time a `1` is found:
  - If a previous `1` exists, compute the distance.
  - Update the maximum distance.

---

### Complexity

- **Time Complexity**: `O(1)` (Maximum 31 iterations regardless of input)
- **Space Complexity**: `O(1)`

---

## 🚀 Approach 2 — Dynamic Bit Shifting (Recommended)

### 💡 Idea

Instead of scanning fixed bit positions, we:

1. Check the least significant bit using `n & 1`.
2. Track the position of each bit.
3. Right shift `n` using `n >>= 1`.
4. Continue until `n` becomes `0`.

This avoids hardcoding a bit limit and scales naturally.

---

### Code (TypeScript)

```ts
function binaryGap(n: number): number {
    let maxDist = 0;
    let last = -1;
    let position = 0;

    while (n > 0) {
        if (n & 1) {
            if (last !== -1) {
                maxDist = Math.max(maxDist, position - last);
            }
            last = position;
        }

        n >>= 1;
        position++;
    }

    return maxDist;
}
```
---

### Why It Works

- `n & 1` checks whether the current (rightmost) bit is `1`.
- `n >>= 1` shifts bits to the right, effectively moving to the next bit.
- `position` keeps track of the current bit index.
- Every time we encounter a `1`, we compare its position with the previous `1`.

---

### Complexity

- **Time Complexity**: `O(log n)` (We process one bit per iteration.)
- **Space Complexity**: `O(1)`

---

### Why This Is Preferred

- No fixed bit limit.
- Cleaner logic.
- More scalable.
- Easier to explain in interviews.

---

## 📊 Comparison of Approaches

| Criteria | 🛠 Fixed 31-Bit Scan | 🚀 Dynamic Bit Shifting |
|------------|----------------------|--------------------------|
| Bit Limit | Hardcoded (0–30) | No hardcoded limit |
| Scalability | Limited to 32-bit assumption | Fully scalable |
| Runtime | O(1) | O(log n) |
| Space | O(1) | O(1) |
| Interview Friendliness | Good | Excellent |
| Code Cleanliness | Moderate | Cleaner |

---

### 🏆 Final Recommendation

Both approaches are optimal and correct.

However:

- If constraints guarantee fixed 32-bit integers → Approach 1 is perfectly fine.
- For cleaner, scalable, interview-quality code → **Approach 2 is preferred**.

Approach 2 avoids assumptions and models the bit traversal more naturally.
