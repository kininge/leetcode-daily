# 🧩 Problem #3532 - Path Existence Queries in a Graph I

## 💡 Intuition

At first glance, the graph appears to be dense.

For every pair of nodes `(i, j)`, an edge exists if

```text
|nums[i] - nums[j]| <= maxDiff
```

A straightforward solution is to compare every pair and build the graph.

```text
For every i
    Compare with every j > i
```

However, with

```text
n = 10^5
```

this becomes

```text
O(n²)
```

which is too slow.

The key observation is that **`nums` is already sorted**.

Because of the sorted order, we only need to connect **adjacent elements** that satisfy the condition.

## 🔍 Key Observation

Suppose

```text
nums

2 5 6 8
```

and

```text
maxDiff = 2
```

Edges are

```text
5 -- 6 -- 8
```

Although

```text
5 ------ 8
```

also satisfies the condition, adding this edge does **not** change connectivity.

The existing path

```text
5 -> 6 -> 8
```

is already sufficient.

Therefore, connecting only adjacent nodes preserves every connected component.

Whenever

```text
nums[i] - nums[i-1] <= maxDiff
```

we simply union

```text
i-1
```

and

```text
i
```

## ✅ Solution

```ts
function pathExistenceQueries(n: number, nums: number[], maxDiff: number, queries: number[][]): boolean[] {
    const unionFind = new MyUnionFind(n);

    // Union only adjacent nodes.
    for (let i = 1; i < n; i++) {
        const diff = Math.abs(nums[i] - nums[i - 1]);

        if (maxDiff >= diff) {
            unionFind.union(i, i - 1);
        }
    }

    const ans = [];

    for (const [u, v] of queries) {
        ans.push(unionFind.connected(u, v));
    }

    return ans;
}

class MyUnionFind {
    private _size: number[];
    private _root: number[];
    private _components: number;

    constructor(nodesInGraph: number) {
        this._components = nodesInGraph;
        this._size = new Array(nodesInGraph).fill(1);
        this._root = [];

        for (let i = 0; i < nodesInGraph; i++) {
            this._root.push(i);
        }
    }

    find(node: number): number {
        if (node === this._root[node]) {
            return node;
        }

        const root = this.find(this._root[node]);
        this._root[node] = root;

        return root;
    }

    union(node1: number, node2: number): boolean {
        const rootNode1 = this.find(node1);
        const rootNode2 = this.find(node2);

        if (rootNode1 === rootNode2) {
            return false;
        }

        const unionSize1 = this._size[rootNode1];
        const unionSize2 = this._size[rootNode2];

        if (unionSize1 >= unionSize2) {
            this._size[rootNode1] += unionSize2;
            this._root[rootNode2] = rootNode1;
        } else {
            this._size[rootNode2] += unionSize1;
            this._root[rootNode1] = rootNode2;
        }

        this._components--;

        return true;
    }

    connected(node1: number, node2: number): boolean {
        return this.find(node1) === this.find(node2);
    }

    componentSize(node: number): number {
        return this._size[this.find(node)];
    }

    components(): number {
        return this._components;
    }
}
```

---

## ⚖️ Why it Works

Since `nums` is sorted, every connected component forms a **contiguous range of indices**.

If two adjacent elements satisfy

```text
nums[i] - nums[i-1] <= maxDiff
```

then they belong to the same component.

By repeatedly unioning adjacent elements, every node within the same contiguous range becomes part of a single connected component.

Adding edges between non-adjacent nodes cannot merge additional components because those nodes are already connected through intermediate adjacent nodes.

Therefore, after preprocessing:

```text
find(u) == find(v)
```

is true **if and only if** there exists a path between `u` and `v`.

## ⏱ Complexity

Let

```text
n = number of nodes
q = number of queries
```

### Time

Building connected components

```text
O(n · α(n))
```

Processing queries

```text
O(q · α(n))
```

Overall

```text
O((n + q) · α(n))
```

where `α(n)` is the inverse Ackermann function and is effectively constant.

---

### Space

```text
O(n)
```

for the Union-Find data structure.

## 🏆 Key Takeaways

* Connectivity problems with many queries are strong candidates for **Union-Find**.
* When data is **sorted**, avoid constructing every possible edge.
* Sometimes the goal is to preserve **connected components**, not every edge.
* Adjacent edges can be sufficient to represent the entire graph.
