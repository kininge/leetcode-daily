# 🧩 Problem #1871 – Jump Game VII

**Difficulty:** Medium   
**Topics:** `Dynamic Programming`, `Sliding Window`  
**Link:** [https://leetcode.com/problems/jump-game-vii/](https://leetcode.com/problems/jump-game-vii/)

---

## 📝 Problem Summary

You are given: `s`

which is a binary string.

You start at: `index 0`

You may jump from index `i` to index `j` if: `i + minJump <= j <= i + maxJump`

AND: ` s[j] === '0'`

Return: `true`

if last index is reachable.

Otherwise: `false`

---

## 💡 Intuition

For every index: `i`

we can jump to a RANGE of indices: `[i + minJump, i + maxJump]`

A brute-force DFS or graph traversal would repeatedly scan overlapping ranges.

The key optimization is:

```txt
Instead of simulating jumps,
track whether ANY reachable index exists
inside valid jump window.
```

This naturally leads to: `Sliding Window + DP`

---

## 🛠 Approach — Sliding Window Reachability DP

* We can tell index `i` is reachable. If we know in range `[i-maxJump, i-minJump]` has any index 'j' which is `0` and reachable.
* What if we store reachable `0` indexs count for previous window range `[i-maxJump, i-minJump]`

*Example:* 
```txt
  s = "00111010"
  minJump = 3
  maxJump = 5
```

* Create DP `array` to store rechable indexes and varible to store count of reachable `0` indexes in last valid window.
* Mark `s[0]` as `true` as starting point


## 💻 Code (TypeScript)

```ts
function canReach(
    s: string,
    minJump: number,
    maxJump: number
): boolean {

    const n = s.length;

    // edge case
    if (s[n - 1] === '1') return false;

    // approach: maintain reachable indexes count
    // by help of sliding window

    // initially consider all indexes unreachable
    const reachableIndexes =
        new Array(n).fill(false);

    // because we know 0th index is always reachable
    reachableIndexes[0] = true;

    let reachableIndexCount = 0;

    for (let i = 1; i < n; i++) {

        // window for ith index
        const rangeStartIndex = i - maxJump;
        const rangeEndIndex = i - minJump;

        // remove leaving index
        if (
            (rangeStartIndex - 1) >= 0 &&
            reachableIndexes[rangeStartIndex - 1]
        ) {
            reachableIndexCount--;
        }

        // add entering index
        if (
            rangeEndIndex >= 0 &&
            reachableIndexes[rangeEndIndex]
        ) {
            reachableIndexCount++;
        }

        // current index reachable?
        if (
            reachableIndexCount > 0 &&
            s[i] === '0'
        ) {
            reachableIndexes[i] = true;
        }
    }

    return reachableIndexes[n - 1];
}
```

## ✅ Why This Works

At every index:

```txt
reachableIndexCount > 0
```

means:

```txt
there exists at least one reachable position
inside valid jump range
```

So current index becomes reachable.

Sliding window guarantees:

```txt
constant-time range updates
```

which reduces complexity from:

```txt
O(n²)
```

to:

```txt
O(n)
```

## 📌 Key Takeaways

* Repeated scanning of overlapping ranges is a strong signal for:

```txt
Sliding Window
```

* Many graph problems can be converted into:

```txt
Range DP
```

* Instead of simulating all jumps explicitly:

```txt
track whether a valid jump source exists
```

* Sliding windows are excellent for:

```txt
"Does any valid state exist in this range?"
```

queries.

## ⚠️ Common Mistakes

### 1️⃣ Repeatedly Rescanning Same Range

Overlapping ranges should trigger:

```txt
Sliding Window thinking
```

---

### 2️⃣ Explicitly Simulating Every Jump

That leads to:

```txt
O(n²)
```

---

### 3️⃣ Forgetting Window Maintenance

Sliding windows require:

* removing old index
* adding new index

carefully.

---

### 4️⃣ Forgetting To Check Current Position

Even if jump is possible:

```txt
s[i] must be '0'
```

otherwise landing is invalid.
