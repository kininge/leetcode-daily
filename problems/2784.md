# 🧩 Problem #2784 – Check if Array is Good

**Difficulty:** Easy    
**Topics:** `Array`, `Hash Map`, `Sorting`     
**Link:** [https://leetcode.com/problems/check-if-array-is-good/](https://leetcode.com/problems/check-if-array-is-good/)

---

## 📝 Problem Summary

An array is considered good if it is a permutation of:

```txt
base[n] = [1, 2, 3, ..., n-1, n, n]
```

Meaning:

* numbers from `1` to `n-1` appear exactly once
* number `n` appears exactly twice
* total length becomes `n + 1`

Given an array `nums`, return:

```txt
true
```

if the array is good, otherwise return:

```txt
false
```

---

### Key Observation

If:

```txt
nums.length = m
```

then the expected good array must be:

```txt
[1,2,3,...,m-1,m-1]
```

because:

```txt
base[n]
```

has length:

```txt
n + 1
```

So for array length:

```txt
m
```

candidate value becomes:

```txt
n = m - 1
```

---

### Constraints

* `1 <= nums.length <= 100`
* `1 <= nums[i] <= 200`

---

## 💡 Intuition

The array must satisfy two conditions:

1. Every number from:

```txt
1 → n-1
```

must appear exactly once.

2. Number:

```txt
n
```

must appear exactly twice.

Since:

```txt
n = nums.length - 1
```

we can validate the array using:

* frequency counting
* sorting

Both approaches work efficiently because constraints are small.

---

## 🛠 Approach 1 — HashSet Validation

### Idea

Use a `Set` to track numbers already seen.

Rules:

* maximum value (`n-1`) must appear exactly twice
* all other values must appear exactly once
* numbers from `1` to `n-2` must all exist

---

### Step 1 — Determine Expected Maximum

If array length is:

```txt
n
```

then expected duplicated value becomes:

```txt
n - 1
```

---

### Step 2 — Track Frequencies

For every number:

* if it equals `n-1`

  * allow exactly two occurrences
* otherwise

  * duplicates are invalid

---

### Step 3 — Verify Missing Numbers

Ensure every number from:

```txt
1 → n-2
```

exists exactly once.

---

### Code (TypeScript)

```ts
function isGood(nums: number[]): boolean {
    const n = nums.length;

    const seen = new Set<number>();

    const maxValue = n - 1;

    let maxValueCount = 0;

    for (const num of nums) {

        if (num < 1 || num > maxValue) {
            return false;
        }

        if (num === maxValue) {

            maxValueCount++;

            if (maxValueCount > 2) {
                return false;
            }

            seen.add(num);
        }
        else if (seen.has(num)) {
            return false;
        }
        else {
            seen.add(num);
        }
    }

    for (let i = 1; i < maxValue; i++) {
        if (!seen.has(i)) {
            return false;
        }
    }

    return maxValueCount === 2;
}
```

## 🛠 Approach 2 — Sorting Validation

### Idea

After sorting, a valid good array must look exactly like:

```txt
[1,2,3,...,n-1,n-1]
```

So we simply verify:

* first `n-1` elements follow increasing sequence
* last element equals `n-1`

---

### Step 1 — Sort Array

```ts
nums.sort((a, b) => a - b)
```

---

### Step 2 — Verify Sequence

Expected values:

| Index | Expected Value |
| ----- | -------------- |
| `0`   | `1`            |
| `1`   | `2`            |
| `2`   | `3`            |
| `...` | `...`          |
| `n-2` | `n-1`          |
| `n-1` | `n-1`          |

---

### Why It Works

A good array is simply a permutation of:

```txt
[1,2,3,...,n-1,n-1]
```

Sorting converts every permutation into the same ordered representation.

So validation becomes straightforward.

---

### Code (TypeScript)

```ts
function isGood(nums: number[]): boolean {
    nums.sort((a, b) => a - b);

    const n = nums.length;

    for (let i = 0; i < n - 1; i++) {

        if (nums[i] !== i + 1) {
            return false;
        }
    }

    return nums[n - 1] === n - 1;
}
```

## ⚖️ Approach Comparison

| Approach           | Time Complexity | Auxiliary Space                    | Notes                                                |
| ------------------ | --------------- | ---------------------------------- | ---------------------------------------------------- |
| HashSet Validation | `O(n)`          | `O(n)`                             | Better asymptotic complexity but more logic handling |
| Sorting Validation | `O(n log n)`    | `O(1)` *(excluding sorting space)* | Cleaner and easier to explain                        |

---

## 🏆 Which Approach Is Better?

For this problem:

```txt
n <= 100
```

So readability matters more than asymptotic optimization.

The sorting solution is often preferable because:

* implementation is shorter
* logic is easier to verify
* debugging is simpler
* interview explanation becomes cleaner

## 📌 Key Takeaways

* The most important observation is:

```txt
If array length = n
then expected duplicated value = n-1
```

* Sorting can simplify permutation-validation problems significantly.

* Better asymptotic complexity does not always produce the cleanest solution.

* For small constraints, readability and maintainability often matter more.

## ⚠️ Common Mistakes

### 1️⃣ Forgetting Array Length Relationship

If:

```txt
nums.length = n
```

then expected array becomes:

```txt
[1,2,...,n-1,n-1]
```

NOT:

```txt
[1,2,...,n,n]
```

---

### 2️⃣ Allowing More Than Two Maximum Values

The largest value must appear:

```txt
exactly twice
```

not:

```txt
at least twice
```

---

### 3️⃣ Forgetting Missing Number Validation

Even if duplicates look correct, every value from:

```txt
1 → n-2
```

must exist.

---

### 4️⃣ Ignoring Invalid Values

Values outside:

```txt
1 → n-1
```

cannot appear in a good array.
