# 🧩 Problem #1404 – Number of Steps to Reduce a Number in Binary Representation to One

**Difficulty:** Medium  
**Topics:**  `String`, `Simulation`, `Bit Manipulation`, `Greedy`  
**Link:** https://leetcode.com/problems/number-of-steps-to-reduce-a-number-in-binary-representation-to-one/

---

## 📝 Problem Summary

You are given a binary string `s` representing a positive integer.

You must repeatedly apply the following rules until the number becomes `1`:

1. If the number is **even** → divide it by 2.
2. If the number is **odd** → add 1 to it.

Return the total number of steps required.

---

### Example

Input:  `s = "1101"`  

Steps: 
```txt
1101 (13) → odd → +1 → 1110
1110 (14) → even → /2 → 111
111 (7) → odd → +1 → 1000
1000 (8) → even → /2 → 100
100 (4) → even → /2 → 10
10 (2) → even → /2 → 1
```

Total steps = `6`

---

### Constraints

- `1 <= s.length <= 500`
- `s[i]` is `'0'` or `'1'`
- `s[0] == '1'` (no leading zeros)

---

### Key Observation

- Even check → last bit is `0`
- Odd check → last bit is `1`
- Binary operations (divide by 2, add 1) can be simulated using bit logic

The challenge is to perform these operations efficiently without repeatedly rebuilding large strings.

---

## 💡 Intuition

We simulate two operations:

- If last bit is `0` → divide by 2 → remove last bit
- If last bit is `1` → add 1 → perform binary addition

A direct simulation would:
- Rebuild the string for addition
- Remove last character for division
- Repeat until `"1"`

This works, but rebuilding strings repeatedly is expensive.

---

### Key Binary Insight

Instead of modifying the entire string each time, observe:

When scanning from **right to left**, we only care about:

- The current bit
- Whether there is a carry from previous addition

Why?

Because adding 1 to a binary number behaves like: 
```txt
    0111
  +    1
  ------
    1000
```

It flips trailing `1`s to `0`s until it finds a `0`.

So instead of physically modifying the string:

We can simulate the process by:

1. Iterating from right to left
2. Tracking a `carry`
3. Counting operations based on effective bit value

---

### Core Idea for Optimization

For each bit (from right to left, excluding MSB):

Let: `effectiveBit = currentBit + carry`

- If `effectiveBit` is even → one operation (division)
- If `effectiveBit` is odd → two operations (add + divide)

We continue until we reach the most significant bit.

This avoids:
- Rebuilding strings
- Re-parsing binary values
- Repeated O(n) operations

And gives a linear-time solution.

---

## 🛠 Approach 1 — Full String Simulation (Brute Force)

### Idea

Simulate the problem exactly as described:

- If number is odd → perform binary addition (`+1`)
- If number is even → divide by 2 (remove last bit)

We repeatedly modify the string until it becomes `"1"`.

This is a literal implementation of the rules.

---

### Why it works

- Binary odd check → last character is `'1'`
- Binary even check → last character is `'0'`
- Division by 2 in binary → remove last bit
- Addition requires handling carry propagation from right to left

The logic is correct but involves repeated string rebuilding.

---

### Code (TypeScript)

```ts
function numSteps(s: string): number {
    // O(n)
    const addOneInNumber = (num: string) => {
        let newNum:string = '';
        let carry:number = 1;

        // O(n)
        for(let i=num.length-1; i>=0; i--){
            if(carry === 1) {
                if(num[i] === '1') newNum = "0" + newNum;
                else {
                    carry = 0;
                    newNum = "1" + newNum;
                }
            } else newNum = num[i] + newNum;
        }
        if(carry === 1) newNum = "1" + newNum;

        return newNum;
    }
    // O(n)
    const divideByTwo = (num: string) => {
        return num.substring(0, num.length-1); // left shift opration equivalent
    }

    let num:string = s;
    let count:number = 0;
    // O(n^2)
    while(num !== '1'){
        // odd number case
        if(num[num.length-1] === '1') num = addOneInNumber(num);
        // even case
        else num = divideByTwo(num);

        count++;
    }

    return count;
}
```

### Complexity

- **Time Complexity:**
  - Each addition: `O(n)`
  - Each division (substring): `O(n)`
  - In worst case, up to `O(n)` operations
  - Overall: `O(n²)`
- **Space Complexity:**
  - New strings created repeatedly → `O(n)`

## 🚀 Approach 2 — Single Pass with Carry Simulation (Optimal)

### Idea

Instead of rebuilding the string repeatedly, observe:

- Even number (`...0`) → one step (divide by 2)
- Odd number (`...1`) → one step to add 1, then one step to divide

Binary addition only affects trailing `1`s due to carry propagation.

So we:

1. Traverse the string from right → left
2. Maintain a `carry`
3. Count operations based on the effective bit value

We do NOT modify the string.

---

### Why it works

Let: `effectiveBit = (currentBit) + carry`


Cases (except MSB):

- If `effectiveBit` is **0**  
  → one operation (division)

- If `effectiveBit` is **1**  
  → two operations:
     - add 1
     - divide by 2  
  → and set `carry = 1`

We process all bits except the most significant bit.

After finishing, if `carry == 1`, we need one extra step.

---

### Code (TypeScript)

```ts
function numSteps(s: string): number {
    let steps = 0;
    let carry = 0;

    // Traverse from right to left (excluding MSB)
    for (let i = s.length - 1; i > 0; i--) {
        const bit = Number(s[i]);
        const effective = bit + carry;

        if (effective === 1) {
            // odd → add 1 + divide
            steps += 2;
            carry = 1;
        } else {
            // even → divide only
            steps += 1;
        }
    }

    // If carry remains after processing MSB
    return steps + carry;
}
```

### Complexity

- **Time Complexity:**
  - Single pass through string → `O(n)`
- **Space Complexity:**
  - Only constant variables used → `O(1)`

### Why this is better than Approach 1

- No string rebuilding
- No repeated substring operations
- No repeated full binary additions
- Strictly linear time
- Cleaner and more interview-ready

This is the optimal solution.

---

## 📊 Comparison of Approaches

| Criteria | 🛠 Approach 1 — Full String Simulation | 🚀 Approach 2 — Single Pass with Carry |
|-----------|----------------------------------------|----------------------------------------|
| Core Idea | Physically simulate +1 and /2 on string | Simulate logically using carry |
| Addition Handling | Rebuild full string with carry propagation | Track carry in O(1) |
| Division Handling | Create substring | Count step only |
| Time Complexity | O(n²) | O(n) |
| Space Complexity | O(n) (new strings repeatedly) | O(1) |
| Constant Factor | High | Very Low |
| Implementation Complexity | Longer & error-prone | Short & clean |
| Interview Strength | Basic simulation | Strong binary reasoning |

---

### 🏆 Final Recommendation

- Approach 1 correctly simulates the problem but performs unnecessary string rebuilding.
- Approach 2 leverages binary behavior and carry propagation to achieve optimal efficiency.

For interviews and large inputs,  
**Approach 2 (Single Pass Carry Simulation) is strongly preferred.**
